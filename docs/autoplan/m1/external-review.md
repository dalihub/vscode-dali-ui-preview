# M1 External Review (independent)

> Reviewer formed a fresh opinion from the M0→M1 diff (`git diff 65693ed..4af3a4e`), the actual dali-ui headers, the parser, and by **re-running** `npm run test:e2e:server` from a clean source recompile **plus a mutation test**. ADR/architecture docs were intentionally not read.

---
## Verdict: PASS

The 6 F1.x promises hold at the code level, the server code paths are genuinely reached, and the goldens are constructed so that a regression to the M0 buggy behavior actually FAILS (empirically proven by mutation). Scope is clean. The honesty claims (loud magenta, render-only for F1.3 pixels) are truthfully implemented. Two real-but-minor gaps keep this from being a flawless pass: `SetVisibility` ships with zero test coverage, and F1.3's "sized box" is only verified by render+metadata, never by a pixel golden.

## Findings

- **Finding 1 — Goldens are NOT test-theatre; a regression genuinely fails (mutation-proven).** I temporarily stubbed out `else if (n == "SetCornerRadius")` in `docker/preview_server.cpp:564`, re-ran `npm run test:e2e:server` (local backend recompiles the server from source, so the change took effect), and corner-radius rendered SQUARE and **FAILED its golden: 1.99% pixels differ (3184/160000)**, above the `match = diffPercent < 0.01` threshold (`test/e2e/imageComparator.ts:89`). I then restored the file byte-for-byte (`git diff 4af3a4e -- docker/preview_server.cpp` is empty). This directly answers the central "could a golden have captured a wrong render that passes against itself" question: no — the old buggy behavior is distinguishable and fails. Why it matters to a user: the corner-radius fix is real and guarded against future regression.

- **Finding 2 — All six setters reach the correct path; parser↔server property strings match exactly (no off-by-one).** The T1 parser stores every `.Method(args)` verbatim into `node.properties[method]` (`src/cppParser.ts:308`), STRING tokens **retain their surrounding quotes** (`src/cppParser.ts:183`, `text: src.slice(i, j)` spanning both `"`), and a trailing `.WithAlpha(...)` is preserved as one nested chain on the call result (`src/cppParser.ts:425-434`). So the server receives exactly `"Method Text"` (quotes intact → stripped once at `preview_server.cpp:646` / Label method-form), `"poster.jpg"` (stripped once for ImageView), `Color::RED` (matches the named table), and `UiColor(0x00d4a8).WithAlpha(0.5f)` (single string → `SBParseUiColor` strips `.WithAlpha`). Every setter signature exists in the headers: `View::SetCornerRadius(float)` (view.h:867, single-arg overload correctly chosen), `SetOpacity` (405), `SetVisibility(bool)` (389), `SetBorderlineWidth` (960), `SetBorderlineColor(const UiColor&)` (979), `Label::SetText` (200) / `SetMarkupEnabled` (414), `ImageView::SetResourceUrl` (176). No dead branches. Why it matters: the "silently wrong" classes (square / black / empty / bare ImageView) are each actually addressed, not just claimed.

- **Finding 3 — `.WithAlpha` parsing is correct, not a silent no-op (correctness trap cleared).** `SBParseUiColor` (`docker/preview_server.cpp:438`) finds `.WithAlpha(`, parses `0.5f` via `SBParseFloat` (handles the trailing `f`), strips it to leave `UiColor(0x00d4a8)`, resolves the hex (the newly-added `base.back()==')'` guard correctly lets `Color::RED` fall through to the named table instead of mis-entering the hex branch), then `result = result.WithAlpha(0.5)`. `UiColor(uint32_t rgb, float a=1.0f)` (ui-color.h:111, param literally named `rgb`) confirms a 6-hex value is RGB with implied alpha 1.0, and `WithAlpha` (ui-color.h:167) returns a new color with replaced alpha. The golden visually confirms this: the `.WithAlpha(0.5)` teal box is the SAME desaturated teal as the 0.5-opacity box in opacity-borderline — i.e. alpha 0.5 was applied, not ignored. Why it matters: the `.WithAlpha` half of F1.5 is genuinely live.

- **Finding 4 — Loud-magenta fallback is real (honesty claim verified).** `docker/preview_server.cpp` ends `SBParseUiColor` with `if (!resolved) result = UiColor(1.0f, 0.0f, 1.0f, 1.0f);` — magenta, not black. This matches the spec's "unknown non-hex token → loud magenta (not black)" and the exec-validation self-check. Why it matters: a future unsupported color token shows up as a visible bug, not a silent black box (the exact failure-mode M1 set out to kill).

- **Finding 5 (gap) — `SetVisibility` ships with ZERO test coverage.** F1.6 / the checklist promise opacity/**visibility**/borderline, but `grep -rn SetVisibility test/samples/` returns nothing; the only F1.6 sample (`opacity-borderline.preview.dali.cpp`) exercises opacity + borderline only. The `SetVisibility(a0.find("true") != npos)` branch (`preview_server.cpp:566`) is therefore unexecuted by any golden, and its bool parse is a loose substring search (works for the literal `true`/`false` args, but untested and not equality-checked). A hidden-element golden would also be slightly odd (a fully-hidden View renders nothing distinguishable), but a visible→hidden child *pair* would have been a meaningful golden. Why it matters: one of the six advertised properties is unverified; a regression in `SetVisibility` would pass CI silently — the very thing M1 is about.

- **Finding 6 (honesty, acceptable) — F1.3 "sized ImageView" is verified by render+metadata, never by pixels; this is disclosed.** The `// @render-only` short-circuit (`test/e2e/serverGoldenRunner.ts:135`) returns `passed:true` WITHOUT pixel comparison, but only AFTER `result.success` (line 123) and `actualPng` existence (line 126) checks — and a parse-null fails even earlier (line ~97). So it does NOT hide a broken render; it only skips the *pixel* assertion. I confirmed the F1.3 promise objectively from the captured metadata: the method-form `.SetResourceUrl("poster.jpg")` produced `"type":"ImageViewImpl","w":200,"h":200` at (100,100) — a real sized ImageView, not the M0 bare empty View. The async-placeholder-is-non-deterministic justification is plausible and the scoping is honestly reflected in spec.md ("실제 픽셀은 M5") and exec-validation. Why it matters: the escape hatch is narrow and labeled, not a failure-hider — but see "could be stronger" #1.

## Could-be-stronger (even if PASS)
1. **Add a real pixel golden for F1.3 instead of render-only.** In this headless local backend the broken-image placeholder may well be deterministic (the run was reproducible). At minimum, assert the node dimensions from the metadata (w==200,h==200) inside the runner so F1.3 has a *positive automated* check rather than only "rendered to a PNG" — today the only thing distinguishing F1.3 from a bare-View regression is a human reading the metadata, which no test enforces.
2. **Add a `SetVisibility` sample** (e.g. two sibling boxes, one `.SetVisibility(false)`) so the 6th property is actually executed and a pixel golden distinguishes shown-vs-hidden. As written, F1.6's "visibility" rides entirely on code-review, not on the suite.
3. **Harden the `.WithAlpha` parse for robustness**: `base.find(')', open)` takes the first `)` after the marker, which is fine for a float arg but would mis-parse a nested call inside `WithAlpha`. A small comment or a balanced-paren scan would prevent a future foot-gun; likewise `SetVisibility`'s `find("true")` could be a trimmed-equality to avoid matching a stray substring.
